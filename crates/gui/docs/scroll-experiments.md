# Scroll Experiments — Issue #82

Trial records for scroll implementation in GPUI 0.2 (v0.2.2) / Taffy v0.9.0.

## Prerequisites

### GPUI 0.2 Overflow API

Generated by the `overflow_style_methods!()` macro in `gpui-macros` (`gpui-macros/src/styles.rs`):

| Method | Effect |
|----------|------|
| `overflow_hidden()` | `overflow.x = Hidden`, `overflow.y = Hidden` |
| `overflow_x_hidden()` | `overflow.x = Hidden` |
| `overflow_y_hidden()` | `overflow.y = Hidden` |
| `overflow_scroll()` | `overflow.x = Scroll`, `overflow.y = Scroll` |
| `overflow_x_scroll()` | `overflow.x = Scroll` |
| `overflow_y_scroll()` | `overflow.y = Scroll` |

### Taffy Version Differences (Important)

| Project | GPUI | Taffy | Horizontal Scroll |
|-------------|------|-------|-------------|
| **Zelkova** | 0.2.2 (crates.io) | **0.9.0** (pinned with `"=0.9.0"`) | Not possible |
| **Zed** | 0.2.2 (git fork) | **0.10.1** (`"=0.10.1"`) | Possible |

It is likely that overflow handling improved between Taffy 0.9.0 and 0.10.1.
GPUI 0.2.2 pins Taffy to `"=0.9.0"`, so upgrading Taffy alone is not possible.

### GPUI content_size Calculation

`div.rs:1371-1394`: scroll range = `content_size - bounds.size`.

`content_size` is calculated from the min/max of **direct children's layout bounds**:
```rust
for child_layout_id in &request_layout.child_layout_ids {
    let child_bounds = window.layout_bounds(*child_layout_id);
    child_min = child_min.min(&child_bounds.origin);
    child_max = child_max.max(&child_bounds.bottom_right());
}
(child_max - child_min).into()
```

When content_size == bounds.size, scroll range = 0 (no scrolling possible).

### GPUI Horizontal Scroll Wheel Handling

`div.rs:2424-2430`: When `overflow.x == Overflow::Scroll`, adds the wheel's `delta.x` to `scroll_offset.x`.
The event handling on the GPUI side is correctly implemented. The problem is that the scroll range becomes 0.

### Zed's Approach

1. **Editor body**: Custom `Element` trait implementation for custom painting. Horizontal scroll position is constrained via `clamp_scroll_left()`.
2. **UI components**: Uses `div` + `overflow_scroll()`. Works correctly with Taffy 0.10.1.
3. **GPUI scrollable.rs sample**: Nested div confirms vertical and horizontal scrolling.

### Root Cause Analysis of Failures

**Why content_size == bounds.size:**

1. **GPUI's content_size calculation method (primary cause)**: content_size in `div.rs` is calculated from direct children's layout bounds. Even if Taffy sets overflow, GPUI uses the children's actual layout size as-is, so if the container expands, content_size also expands. **This is independent of the Taffy version** (verified in Approach 8).

2. **Flex stretch alignment**: GPUI's div defaults to `Display::Flex`. Children are stretched to container width via `align_items: stretch`. Explicit `w(px(...))` may also be overridden by stretch.

3. **Constraint chain collapse**: Even with `max_w_full()` set, the upstream parent gets pulled wider by its children, causing `max_width` to resolve to a large value.

**Why Zed can scroll horizontally**: Not due to the Taffy version, but because it manages the scroll container itself via a custom `Element` trait implementation (e.g., `clamp_scroll_left()`).

## Solution Considerations

### ~~A. Package Upgrade (recommended)~~ → Rejected

Upgraded Taffy 0.9.0 → 0.10.1, but horizontal scrolling did not work.
(Verified in Approach 8)

The root cause is not the Taffy version but GPUI's `content_size` calculation using children's layout bounds as-is.

### B. Custom Element Implementation (recommended)

Implement GPUI's `Element` trait to manage the scroll container ourselves.

```rust
pub trait Element {
    type RequestLayoutState;
    type PrepaintState;

    fn request_layout(&mut self, ...) -> (LayoutId, Self::RequestLayoutState);
    fn prepaint(&mut self, ..., bounds: Bounds<Pixels>, ...) -> Self::PrepaintState;
    fn paint(&mut self, ..., bounds: Bounds<Pixels>, ...);
}
```

**Design proposal:**

```
EditorContentElement {
    scroll_y: f32,     // Vertical scroll offset
    scroll_x: f32,     // Horizontal scroll offset (when wrap=false)
    lines: Vec<String>,
    // ...
}

request_layout():
    - Determine own size from parent (equivalent to w_full, flex_1)
    - content_size = (max_line_width, total_line_height)
    - Set bounds and content_size on ScrollHandle

paint():
    - Clip within bounds (apply scroll_offset via window.with_element_offset)
    - Render only visible lines (virtualization)
    - Render cursor and selection range within paint as well
```

**Advantages:**
- Does not depend on Taffy's overflow
- Same approach as Zed
- Full control

**Disadvantages:**
- High implementation cost (several days of work)
- Must handle mouse events (click, drag) manually
- Complex IME input integration

### C. Patch GPUI's Scroll Implementation

Fix the `content_size` calculation in `div.rs` to prevent container expansion when overflow is set to scroll.

**Specific proposal:**
- After calculating `scroll_max` in `prepaint()`, fix `bounds.size` to the container's original size
- Or, add a min-width constraint to nodes with `overflow: scroll` during `request_layout()`

Risk: Requires deep understanding of GPUI internals. Potential side effects on other components.

## Recommended Roadmap

1. ~~**Package upgrade**: Taffy 0.10.1 → Rejected (no effect)~~
2. **Custom Element implementation**: Manage the scroll container ourselves using the same approach as Zed (the only viable solution)
3. **Keep Taffy 0.10.1 patch**: May improve layout accuracy

## Zed Custom Element Investigation (2025-05-28)

Zed's editor does not rely on GPUI's `div`-based scrolling. It implements the `Element` trait directly and manages scrolling itself.

### Key Files

- `crates/editor/src/element.rs` — `EditorElement`'s `Element` trait implementation
- `crates/editor/src/scroll_manager.rs` — Scroll state management

### request_layout(): How the Container Doesn't Expand

```rust
// element.rs:9798-9871
fn request_layout(&mut self, ...) -> (LayoutId, EditorRequestLayoutState) {
    let mut style = Style::default();
    style.size.width = relative(1.).into();  // Full parent width (relative)
    style.size.height = relative(1.).into(); // Full parent height (relative)

    // Content size is calculated but not reflected in style
    // → Container does not expand
    window.request_layout(style, None, cx)
}
```

**Important**: Using `relative(1.)` prevents Taffy from being pulled wider by content.
Setting `width: relative(1.)` / `height: relative(1.)` in `Style::default()` makes the layout engine resolve to "full parent size", and content width is ignored.

### scroll_max Calculation (inside prepaint)

```rust
// element.rs:10639-10644
let scroll_max: gpui::Point<ScrollPixelOffset> = point(
    ScrollPixelOffset::from(
        ((scroll_width - editor_width) / em_layout_width).max(0.0),
    ),
    max_scroll_top,
);
```

- `scroll_width`: Maximum width across all lines (content width)
- `editor_width`: Viewport width (obtained from bounds)
- `scroll_max.x = max(0, scroll_width - editor_width)`

This is calculated independently, so it does not depend on GPUI's `content_size`.

### Applying Scroll Offset in paint()

```rust
// element.rs:9380-9381
let mut fragment_origin = content_origin
    + gpui::point(
        Pixels::from(-layout.position_map.scroll_pixel_position.x),
        line_y,
    );
```

The scroll position is applied as a **negative offset** to each fragment's drawing position.
Instead of relying on GPUI's `ScrollHandle`, the offset is managed and applied manually.

### Clipping

```rust
// element.rs:991-992, 10857-10858
window.with_content_mask(Some(ContentMask { bounds }), |window| {
    // Draw only within bounds (clipping)
});
```

Clips to the viewport's bounds via `ContentMask`. Anything outside is not drawn.

### Scroll Wheel Event Handling

```rust
// element.rs:7812-7821
let x = (current_scroll_position.x
    * ScrollPixelOffset::from(glyph_width)
    - ScrollPixelOffset::from(delta.x * scroll_sensitivity))
    / ScrollPixelOffset::from(glyph_width);

let mut scroll_position = point(x, y).clamp(&point(0., 0.), &position_map.scroll_max);
```

Adds the wheel's `delta.x` to the scroll position and clamps to `scroll_max`.
This processing is also executed within `Element`'s `paint()` or an `on_scroll_wheel` handler.

### Visible Line Virtualization

```rust
// Calculate visible line range
let start_row = DisplayRow((scroll_position.y + clipped_top_in_lines).floor() as u32);
let end_row = DisplayRow((scroll_position.y + clipped_top_in_lines + visible_height_in_lines).ceil() as u32);
```

Only computes and renders visible lines based on scroll position. Not rendering all lines provides good performance.

### Mouse Events

Mouse event coordinates are adjusted by the scroll offset:
- Click position → converted to content coordinates via `position - scroll_offset`
- Drag → similarly converted to update the selection range

### Complete Data Flow

```
1. request_layout()
   → style.size = relative(1.) to set container size from parent
   → Content width does not affect layout

2. prepaint()
   → Get editor_width from bounds
   → Calculate scroll_width (maximum line width)
   → scroll_max = scroll_width - editor_width
   → Calculate visible line range (start_row, end_row)

3. paint()
   → Clip to viewport via ContentMask
   → Shift each line's draw position by -scroll_offset
   → Draw cursor and selection range with the same offset

4. on_scroll_wheel
   → Add delta to scroll_position
   → Clamp to scroll_max
   → cx.notify() triggers repaint
```

### Design for Applying to Zelkova

When applying Zed's pattern to Zelkova:

```rust

}

        style.size.height = relative(1.).into();
        let layout_id = window.request_layout(style, None, cx);
        (layout_id, ())
    }

    fn prepaint(&mut self, bounds: Bounds<Pixels>, window: &mut Window, cx: &mut App) -> () {
        // Calculate visible line range
        // Calculate scroll_max
    }

    fn paint(&mut self, bounds: Bounds<Pixels>, _: &mut App) -> () {
        // Clip via ContentMask
        // Draw each line at -scroll_offset
        // Draw cursor, selection, and images with the same offset
    }

```

**Migration steps from the current Zelkova editor render():**

1. Define the `EditorContentElement` struct
2. Implement the 3 methods of the `Element` trait
3. Delegate content line rendering from the `Render` trait's `render()` to `EditorContentElement`
4. Move `on_scroll_wheel` into `EditorContentElement::paint()`
5. Adjust mouse event (click, drag) coordinates by scroll_offset
6. The vertical scroll `ScrollHandle` becomes unnecessary (migrate to self-managed)
7. Apply the same offset to existing highlight and image rendering within `paint()`
